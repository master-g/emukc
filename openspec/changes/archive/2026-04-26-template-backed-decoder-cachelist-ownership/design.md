## Context

The decoder-first cache-list path has three useful pieces already in place:

- `main-decoder/src/pipeline.ts` emits a decoder bundle under `main-decoder/out/resources/`, including `cache_rules.json`, `resource_categories.json`, `resource_id_sets.json`, `audio_resources.json`, and `ui_resources.json`.
- `crates/emukc_bootstrap/src/make_list/source/mod.rs` and the `kcs2/resources/*.rs` generators can build a candidate cache list from the `Rules` strategy and sibling decoder bundle assets.
- `examples/decoder_cachelist_compare.rs` reports baseline recall, candidate-only paths, rule-authored output, fallback-authored output, and migration blockers.

The current comparison state shows the pipeline can reproduce the baseline path set, but still depends on fallback ownership for `8335` candidate paths. The largest residuals are concentrated in template-backed domains: `kcs/sound/kc9998`, furniture normal/movable/scripts/thumbnail families, BGM, map/gauge resources, titlecall, useitem card/card_ resources, and area/world-select UI.

Decoded modules already expose deterministic path formulas for several of these families. Examples from the current decoder output include map thumbnail and task-create map path construction, area sally/airunit resource formulas, and furniture loader formulas. The missing contract is a decoder output model that can say: "this family is generated by this observed template, using these runtime inputs, with this completeness state."

This design affects `main-decoder` and `emukc_bootstrap` implementation paths only. It does not add methods to gameplay traits such as `SortieOps`, `MaterialOps`, or `ShipOps`; it does not modify `_impl` gameplay functions; it does not touch `entity::user`, `entity::profile`, or KCSAPI route handlers.

## Goals / Non-Goals

**Goals:**

- Represent decoder-observed path formulas as structured template-backed resource families with provenance and explicit completeness metadata.
- Let `Rules` generation expand complete template-backed families using existing bootstrap runtime inputs and attribute those paths as rule-authored output.
- Keep partial or unresolved template families visible as fallback territory instead of silently claiming decoder authority.
- Improve comparison output so template-backed residuals explain which family, template, or input binding still blocks migration readiness.

**Non-Goals:**

- Do not switch the default cache-list strategy from `Default` or `Manifest` to `Rules`.
- Do not run JavaScript inside Rust bootstrap generation.
- Do not copy Rust fallback constants, CDN-discovered lists, or generated cache-list output into decoder bundle assets.
- Do not introduce new gameplay trait methods, database migrations, Codex mutation semantics, or KCSAPI route behavior.
- Do not require new Codex fields unless implementation proves an existing runtime input cannot provide a required binding safely.

## Decisions

### D1: Add template-backed family descriptors to decoder bundle assets

`main-decoder` will emit structured descriptors for template-backed families instead of forcing every family into concrete file lists.

Each descriptor should include:

- a stable family key such as `map.thumbnail`, `map.gauge_event`, `furniture.normal`, `bgm.battle`, `sound.kc9998`, or `useitem.card`
- the resource domain and output prefix
- a path template represented as structured segments or named placeholders, not raw JavaScript source
- required runtime inputs such as map entries, furniture IDs, BGM IDs, useitem IDs, or sound bucket IDs
- coverage mode: `complete`, `partial`, or `unresolved`
- provenance pointing back to decoded module IDs and extractor evidence

Rationale: concrete lists are still appropriate for directly enumerated files, but they are a poor fit when `main.js` exposes a reusable formula and membership comes from runtime bootstrap data. Template descriptors preserve decoder authority over path shape without inventing membership.

Alternative considered: keep all path formulas in Rust and only emit family tags from the decoder. Rejected because that would keep semantic ownership in Rust fallback code and would not justify rule-authored attribution.

Alternative considered: flatten every template family into concrete paths during decoder extraction. Rejected because the decoder does not own all runtime input sets, and flattening would tempt the pipeline to backfill from Rust or CDN-derived data.

### D2: Expand templates in Rust through typed input bindings, not a general expression VM

`emukc_bootstrap` will expand decoder templates through a small typed expander that binds approved placeholders to inputs already available in bootstrap generation. Initial adapters should live near the affected generators:

- `crates/emukc_bootstrap/src/make_list/source/kcs2/resources/map.rs`
- `crates/emukc_bootstrap/src/make_list/source/kcs2/resources/gauge.rs`
- `crates/emukc_bootstrap/src/make_list/source/kcs2/resources/furniture.rs`
- `crates/emukc_bootstrap/src/make_list/source/kcs2/resources/bgm.rs`
- `crates/emukc_bootstrap/src/make_list/source/kcs2/resources/unversioned.rs`
- `crates/emukc_bootstrap/src/make_list/source/kcs2/resources/use_item.rs`

The shared bundle loading and attribution plumbing should remain under `crates/emukc_bootstrap/src/make_list/source/mod.rs` or the existing rules/bundle helper modules.

Rationale: Rust must generate the final cache list because it already has the manifest, version, and bootstrap context. A typed expander keeps the decoder bundle declarative and avoids embedding a JavaScript runtime or arbitrary expression language in `emukc_bootstrap`.

Alternative considered: define a generic expression interpreter for template formulas. Rejected because the needed templates are simple path-building formulas, and a generic VM would add safety and maintenance risk without clear value.

### D3: Rule-authored attribution requires both decoder evidence and sufficient runtime inputs

Template-expanded output is rule-authored only when:

- the decoder descriptor is present and marked complete for the relevant family or concrete subset
- the descriptor includes provenance from decoded modules
- all required runtime inputs are available and pass validation for that descriptor
- expansion does not require Rust-only membership constants for the covered set

If any condition fails, generation may still use legacy fallback to preserve baseline recall, but the resulting paths must remain fallback-authored with a residual family label.

Rationale: the comparison report currently measures ownership, not just path existence. The rule-authored label should mean the path shape came from decoder evidence and the membership came from declared runtime inputs, not from hidden Rust fallback lists.

Alternative considered: mark any duplicate path as rule-authored if both decoder and fallback can produce it. Rejected because duplicate path strings can hide the fact that a family still depends on fallback for completeness.

### D4: Keep fallback gates family-aware and residual-aware

Generation should run in ordered stages:

1. decoder concrete lists and existing decoder semantic rules
2. decoder template expansion for complete or concretely covered subsets
3. legacy fallback only for unresolved families or uncovered residual members

For a complete template family, broad fallback for that same family must be suppressed. For a partial family, fallback must remain allowed only for the residual members that the decoder bundle did not prove.

Rationale: the current migration blocker is fallback-authored output, especially broad fallback that runs after a decoder-covered portion. Family-aware gates make ownership accounting honest and reduce duplicated fallback attribution.

Alternative considered: leave fallback behavior unchanged and only improve report labeling. Rejected because that would keep migration blockers even after decoder evidence becomes sufficient.

### D5: Comparison remains the acceptance loop

`examples/decoder_cachelist_compare.rs` should surface template-backed residuals separately from generic fallback prefixes. The report should identify:

- template families expanded as rule-authored
- template families left partial or unresolved
- required runtime inputs that were unavailable
- fallback-authored residual counts by family/domain

Rationale: this change is successful only if `fallback_authored_candidate_count` drops without losing `baseline_only_count = 0`. The report needs enough detail to explain the remaining fallback debt.

Alternative considered: rely on decoder unit tests and Rust generator tests only. Rejected because both can pass while live bundle ownership still fails to improve.

## Risks / Trade-offs

- [Risk] Some decoded formulas may be stable while their membership remains runtime-only. -> Mitigation: model templates separately from input sets and keep membership requirements explicit.
- [Risk] A descriptor marked complete too aggressively could suppress fallback and lose baseline paths. -> Mitigation: require provenance, input validation, and comparison verification before fallback suppression.
- [Risk] Template descriptors can become a second path-rule language. -> Mitigation: keep descriptors declarative, restrict placeholder types, and add typed Rust adapters only for known cache-list domains.
- [Risk] Multiple active decoder changes may touch the same assets. -> Mitigation: keep this phase focused on template descriptors and attribution; concrete UI extraction remains separable from descriptor-backed expansion.
- [Risk] Reported `candidate_only_count` may grow if template input binding over-expands. -> Mitigation: treat baseline recall and candidate-only deltas as verification gates, not just fallback-count reduction.

## Migration Plan

1. Extend decoder asset types and extraction for template-backed family descriptors in `main-decoder`.
2. Sync regenerated decoder bundle assets without changing bootstrap defaults.
3. Add Rust bundle loading and typed expansion for the targeted template domains.
4. Narrow fallback gates for complete or covered template families while preserving residual fallback labels.
5. Regenerate `.data/decoder_rules_compare_current.json` or a successor report and compare fallback ownership against the current `8335` fallback-authored baseline.

Rollback strategy:

- Revert descriptor emission if extraction overstates evidence.
- Revert individual Rust template adapters or fallback gates if a family loses baseline recall.
- Since no default strategy or KCSAPI behavior changes, rollback is limited to the decoder-first candidate path and generated decoder assets.

## Open Questions

- Should template descriptors live inside existing domain assets such as `ui_resources.json` and `audio_resources.json`, or should they move into a dedicated `resource_templates.json` sibling asset?
- Which runtime input source should own each family: decoded bundle IDs, existing bootstrap manifest data, cache-derived metadata, or a combination?
- Should `kc9998` be modeled as a sound-rule template family, a concrete bucket with runtime expansion, or both?
- What threshold should make the later default-switch proposal credible: zero fallback-authored paths overall, or zero fallback-authored paths for decoder-covered families with explicit residual exceptions?
