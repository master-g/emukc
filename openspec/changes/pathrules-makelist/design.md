## Context

The cache make-list system in `emukc_bootstrap` generates lists of game resource paths for the CDN cache. It has two parallel generation paths:

1. **Default/Greedy strategies**: Walk through `kcs2/resources/` modules (`ship.rs`, `slot.rs`, `furniture.rs`, etc.) using hardcoded constants for categories, damage variants, hole lists, and coverage IDs.
2. **Manifest strategy**: Derives paths from `resource_manifest.json` entries resolved via Codex, bypassing the entire kcs/kcs2 chain.

The hardcoded constants in path (1) total over 20 separate lists across 3 files:
- `generate.rs`: `SHIP_DAMAGE_VARIANTS`, `SHIP_STANDARD_CATEGORIES`, `SHIP_FULL_CATEGORIES`, `SLOT_STANDARD_CATEGORIES`, `_SP_REMODEL_CATEGORIES`
- `slot.rs`: `BTXT_FLAT_IDS` (336 entries), `ENEMY_PLANE_MAX_ID`, `CHARACTER_HOLES`
- `ship.rs`: `EVENT_SHIP_HOLES`, `ENEMY_SHIP_HOLES`, `SPECIAL_SHIPS`, `SP_REMODEL_SHIPS`, `SP_REMODEL_MES`, `CARD_ROUNDS`, `REWARDS`

The main-decoder already processes the game client code and can extract all of this data. A v2 manifest with a `pathRules` block can replace these constants, making the Default/Greedy paths data-driven.

**Key constraint**: The existing Manifest strategy is intercepted at `source/mod.rs:30` before the kcs/kcs2 chain, so `slot.rs`/`ship.rs` never see Manifest. This design does NOT change that dispatch — it feeds `pathRules` as a data source into the Default/Greedy paths.

## Goals / Non-Goals

**Goals:**
- main-decoder emits `pathRules` in v2 `resource_manifest.json` covering all constants listed above
- Rust `PathRules` type deserializes `pathRules` (optional, backward-compatible with v1)
- Default/Greedy code paths read from `pathRules` when available, fall back to constants for v1
- `has_btxt_flat_coverage()` uses manifest-derived `HashSet<i64>` with constant fallback
- Validation test confirms `pathRules` values match existing constants

**Non-Goals:**
- Changing the strategy dispatch or CLI flags
- Replacing Default/Greedy with Manifest
- Removing the Manifest strategy path
- Covering BGM, furniture, map, gauge, use_item, unversioned resources via `pathRules` (future scope)
- Runtime resource prefetching
- Modifying the Codex loading pipeline

## Decisions

### 1. `pathRules` as data source for Default/Greedy, not strategy replacement

**Decision**: Feed `pathRules` into existing Default/Greedy code paths via `OnceLock`, keeping the strategy dispatch unchanged.

**Rationale**: The Manifest strategy bypasses the entire kcs/kcs2 chain. Ship/slot resource generation in Default/Greedy uses hardcoded constants that are game-version-specific. Replacing those constants with manifest-derived values makes Default/Greedy data-driven without changing their dispatch or output structure.

**Alternative considered**: Merging manifest logic into Default strategy. Rejected — would change existing behavior per OpenSpec decision at `openspec/changes/resource-manifest-rust-integration/design.md:46-52`.

### 2. `OnceLock` for cross-module access

**Decision**: Store `PathRules` in a `static PATH_RULES: OnceLock<PathRules>` in `manifest/mod.rs`. Populate at manifest load time. Downstream modules access via a helper function.

**Rationale**: `OnceLock` is process-lifetime, initialized once, no locking overhead after init. The manifest is loaded early in bootstrap. All downstream code can call `path_rules()` to get `Option<&PathRules>` without threading parameters through every function signature.

**Alternative considered**: Passing `&PathRules` as function parameter through the entire call chain. Rejected — would touch every function signature in `ship.rs`, `slot.rs`, `generate.rs`, and their callers.

### 3. `btxt_flat` coverage via pre-computed set

**Decision**: Store resolved btxt_flat coverage as `static BTXT_FLAT_COVERAGE: OnceLock<HashSet<i64>>`, populated from `pathRules.btxtFlatSlotIds` when available. `has_btxt_flat_coverage()` checks this first, falls back to `BTXT_FLAT_IDS`.

**Rationale**: The manifest's `slotMstIdSources` are expression strings that require `ApiManifest` to resolve. The caller in `battle_rules.rs:1095` only has a `slot_id`. A pre-computed set avoids runtime resolution. The decoder emits the curated list directly.

**Alternative considered**: Resolving `slotMstIdSources` at call time. Rejected — over-matches because universal expressions resolve to "all slotitems", not the curated 336-entry set.

### 4. Constants remain as v1 fallback

**Decision**: All existing constants (`BTXT_FLAT_IDS`, `SHIP_DAMAGE_VARIANTS`, etc.) remain in code. When `OnceLock<PathRules>` is not populated (v1 manifest or no manifest), code falls back to constants.

**Rationale**: Backward compatibility. Users with v1 manifests continue working. Constants can be removed after one release cycle once v2 is stable.

## Risks / Trade-offs

**[pathRules drift]** — Game updates may change category lists, damage variants, or hole sets. Mitigation: `pathRules` is regenerated by the decoder. A test compares `pathRules` values against current constants and reports discrepancies.

**[OnceLock initialization order]** — `OnceLock` must be populated before any Default/Greedy make-list generation runs. Mitigation: populate in `source/mod.rs::make()` at the start of the Default/Greedy branch, before calling `kcs::make()` and `kcs2::make()`.

**[Manifest version mismatch]** — v2 manifest may be from a different game version than the Codex. Mitigation: manifest version field checked at load; warn if stale but don't block. Constants provide safety net.
