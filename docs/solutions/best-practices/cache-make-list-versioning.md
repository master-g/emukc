# cache make-list versioning and rollback handling

## Context

`cache make-list` assigns a version to each cache-list item, and `cache
populate` later uses that version to decide whether to (re)download. Two
failure modes recurred before these rules were fixed: `version.json` was served
from local cache (stale, because `NoVersion` caused a return of the old cached
copy) so generated versions lagged behind reality; and version rollbacks (the
on-disk copy is NEWER than the manifest version) were not detected at make-list
time and leaked into populate as spurious "failures" that were retried
pointlessly.

## Guidance

The version generated by `make-list` MUST NOT be lower than the already-cached
version for the same path.

`version.json` MUST be fetched via a **force-remote** request, NOT from local
cache. The root cause of stale versions was `NoVersion` causing the fetcher to
return the old cached copy; force-remote bypasses that.

The **rollback case MUST be detected and handled at the make-list stage**
(substitute the cached version), NOT deferred to the populate stage. Detecting
it earlier means populate never sees a spurious "newer on disk" failure for
that item.

When a category is ABSENT from `version.json`, make-list SHALL use the cached
version if one exists; otherwise it SHALL log a `warn`.

`populate`, upon encountering `InvalidFileVersion` (the typed variant for a
version rollback), SHALL SKIP the item rather than retry it — a version rollback
is not a download failure. (See Related for the populate-side doc.)

The cache (`kache`) MUST expose a public API `get_cached_version(path)` so
make-list can query the already-cached version when deciding whether a
generated version would be a rollback.

## Why This Matters

A make-list version below the cached version causes populate to either skip a
needed update or, worse, overwrite a newer local file with older content.
Serving `version.json` from cache defeats the entire purpose of versioning.
Deferring rollback detection to populate turns a no-op into a retried network
failure, inflating retry counts and confusing the error-classification logic.
Exposing `get_cached_version` is the single lookup that lets make-list make the
rollback decision at the right stage.

## When to Apply

- When modifying how make-list assigns or looks up versions.
- When changing the `version.json` fetch path — confirm it is force-remote.
- When adding a new rollback-like condition — detect it at make-list, not
  populate.

## Examples

make-list generates version "2" for a path whose cached version is "3". The
rollback check (via `get_cached_version`) substitutes "3" so populate sees the
correct version. `version.json` is fetched with force-remote so a stale local
copy never supplies the versions. A category missing from `version.json` falls
back to the cached version or logs `warn`.
