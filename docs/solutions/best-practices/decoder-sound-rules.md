# Decoder kcs/sound rule contract

## Context

Decoder-authored semantic and algorithmic rules drive `kcs/sound/*`
cache-list generation for ship voice families and non-ship sound buckets
(`9997`, `9998`, `9999`). This contract documents what the rules must
preserve and the fallback-disposition controls that govern when legacy Rust
sound generators remain necessary.

## Guidance

**Formula-driven ship voice families.**

- Represent decoder-derived ship voice generation rules as semantic sound
  rules rather than relying on Rust-authored formulas alone.
- When decoded `main.js` modules expose ship voice playback behavior that
  combines a ship voice bucket, ship identity, and voice-id families, emit a
  sound rule preserving the semantic generation model; downstream generation
  must reproduce the reachable `kcs/sound/kc*/` ship voice paths from that
  rule plus existing manifest data.
- When decoded runtime behavior distinguishes special ship voice families
  (repair voices, special-CG voice groups), preserve those subsets
  explicitly; downstream generation must not rely on Rust-only hardcoded
  lists to discover those subsets for covered families.

**Non-ship sound buckets.**

- Represent decoder-derived non-ship `kcs/sound/*` families (`9997`, `9998`,
  `9999`) as sound rules when directly observable in decoded `main.js`. For
  decoder-observable bucket members, the emitted rules must be strong enough
  that Rust-owned bucket generators stop being the primary source for those
  covered members.
- When decoded modules call sound playback against a bucket such as `9998` or
  `9999`, emit rule data preserving the bucket identity and reachable voice
  IDs; downstream generation must produce those sound paths without
  consulting Rust-owned fallback tables for the covered entries.
- When decoded modules use random-choice playback for a bucket with multiple
  possible voice IDs, preserve all reachable voice IDs; the emitted rule must
  not undercount reachable IDs because of the runtime randomization wrapper.
- When decoded modules contain bucket-specific evidence for `kc9997`,
  `kc9998`, or `kc9999`, include the decoder-observable reachable members
  instead of leaving the bucket effectively empty; remaining unproven members
  stay partial or unresolved rather than forcing the entire family back to
  Rust-owned fallback.

**Unresolved families — honesty.**

- Mark sound-rule families partial or unresolved when the decoder cannot
  derive them safely from decoded evidence. Partial status must describe real
  decoder limits, not extractor blind spots for decoder-observable members.
- When coverage cannot be proven complete, mark the rule partial or
  unresolved; downstream generation must treat that family as fallback
  territory.
- When the decoded script exposes a proper subset of reachable members,
  preserve that observed subset even if the family remains partial overall;
  downstream comparison and generation must distinguish the covered subset
  from the residual fallback-owned remainder.

**Fallback disposition declaration.**

- Emit enough coverage state for each sound-rule family so downstream
  generation can decide whether matching legacy sound fallback remains
  necessary.
- When evidence proves complete coverage for a family, identify it as
  complete; downstream generation must skip the matching legacy Rust sound
  fallback generator for that family.
- When evidence proves only part of a family, identify the covered members
  and mark the family partial or unresolved; downstream generation must
  preserve legacy fallback for the unproven remainder.
- When a family contains both decoder-covered and fallback-covered members,
  the metadata must allow downstream comparison to distinguish rule-authored
  output from fallback-authored residual output; decoder-covered members must
  not be regenerated by broad legacy fallback solely to preserve path
  coverage.

## Why This Matters

The random-choice and bucket evidence rules prevent the most common sound
extraction bug: undercounting reachable voice IDs because the runtime wraps
playback in a random picker. The fallback-disposition declaration is what
allows the comparison example to shrink sound residuals accurately — without
it, broad fallback re-expands covered families and hides decoder progress.

## When to Apply

- When extending decoder extraction for a new `kcs/sound/*` family.
- When reviewing whether a legacy Rust sound generator can be narrowed.

## Examples

- Decoded modules call playback against bucket `9998` with voice IDs
  `[1501, 1502, 1503]`: the rule preserves all three, even if the call site
  picks one at random.
- `kc9997` has partial evidence: the rule preserves the observed subset and
  marks the family partial, so fallback covers only the unproven remainder.
