## Why

The map catalog assembly pipeline uses fill-missing merge semantics — when wikiwiki data provides a field, overlay data (from real API captures) cannot override it. This causes persistent data corruption: wikiwiki has wrong `color_no`/`event_id`/`event_kind` for many cells, and overlay's correct values are silently discarded. After re-bootstrap, node colors revert to white, battle nodes get misidentified as safe, and fleets skip fights. The previous one-time migration script approach (commit 7e0d397) is overwritten on each bootstrap.

A field-authority model is needed: each data field has a designated authoritative source. The merge uses "last non-zero wins" for metadata fields, and assembly order determines authority — higher-priority sources merge last.

## What Changes

- Change merge semantics in `merge_cells()` for metadata fields (`color_no`, `event_id`, `event_kind`, `boss_cell_no`) from fill-missing to "last non-zero wins". Routing fields (`next_cells`, `node_label`, `routing_rules`, `enemy_fleets`, `ship_drops`) remain fill-missing.
- Change assembly order to: wikiwiki → overlay → stat. This way overlay's `color_no`/`boss_cell_no` override wikiwiki's wrong values, and stat's `event_id`/`event_kind` (highest authority) override overlay's color-inferred values.
- **Remove kc_data map source** from the assembly pipeline. kc_data YAML provides node labels, simple route topology, boss flags, and inferred color/event values — all of which are redundantly covered by wikiwiki (labels, routing rules, enemies, drops), overlay (color_no, boss_cell_no), and stat.json (event_id/event_kind). Removing it simplifies the pipeline to three non-overlapping authority sources.
- Complete overlay capture: add `boss_cell_no` extraction from real API data, and `infer_event_from_color` in overlay merge.
- Integrate kcs2-mapdata `stat.json` as highest-authority source for `event_id`/`event_kind`.
- Add full source plumbing for stat: new fields in `ResolvedMapSources`, `MapCatalogBuildReport`, download/cache/fallback logic.
- Regenerate overlay static asset (`assets/public_map_catalog_overlays.json`) after code changes so bootstrap picks up new boss_cell_no and event inference.

## Capabilities

### New Capabilities

- `map-data-authority`: Per-field authority model for map catalog assembly. Authority is determined by merge order (last non-zero wins for metadata fields). Defines source plumbing for kcs2-mapdata `stat.json`, label-matching strategy, and failure mode handling.

### Modified Capabilities

- `sortie`: Map cell metadata (`color_no`, `event_id`, `event_kind`, `boss_cell_no`) will now be authoritative from real game data rather than wiki-parsed approximations. This changes the behavioral contract: cells will always have correct battle/safe/boss classification matching the real game.

## Impact

- **Code**: `crates/emukc_model/src/codex/map/merge.rs` (merge semantics), `crates/emukc_bootstrap/src/map_overlay/capture.rs` (boss capture), `crates/emukc_bootstrap/src/map_overlay/merge.rs` (event inference), `crates/emukc_bootstrap/src/map_pipeline/sources.rs` (remove kc_data, add stat), `crates/emukc_bootstrap/src/map_pipeline/report.rs` (remove kcdata, add stat fields), `crates/emukc_bootstrap/src/map_pipeline/assemble.rs` (simplify to wikiwiki → overlay → stat), `crates/emukc_bootstrap/src/map_pipeline/kcdata.rs` (remove or archive)
- **Data**: `crates/emukc_bootstrap/assets/public_map_catalog_overlays.json` must be regenerated (adds boss_cell_no, inferred event_id/event_kind)
- **Bootstrap**: New download step for kcs2-mapdata `stat.json` from GitHub, cached locally
- **Tests**: Existing merge test needs updated expectations; new tests for stat label matching, stat fallback, overlay regeneration

## Non-goals

- Fixing map routing rules (rules are correct; only cell metadata is wrong)
- Supporting maps without any real data source (event maps with no overlay or stat coverage keep wikiwiki data)
- Changing the Codex load path or runtime behavior
- Adding new gameplay traits or KCSAPI handlers
- Making stat.json or overlay sources dynamically fetched at runtime — both are repo-tracked assets regenerated by CLI commands
