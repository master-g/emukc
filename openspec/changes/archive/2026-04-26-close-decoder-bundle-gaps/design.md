## Context

The decoder-first cache-list pipeline is already structurally in place:

- `main-decoder/src/pipeline.ts` emits a decoder bundle containing `cache_rules.json`, `resource_categories.json`, `resource_id_sets.json`, `audio_resources.json`, and `ui_resources.json`.
- `crates/emukc_bootstrap/src/make_list/source/mod.rs` already supports `CacheListMakeStrategy::Rules` and threads an explicit decoder bundle through generation.
- `examples/decoder_cachelist_compare.rs` already measures rule-authored versus fallback-authored output and marks migration readiness.

The current bottleneck is bundle quality, not bundle shape.

Today:

- `.data/decoder_rules_compare_current.json` reports `baseline_only_count = 0`, `candidate_only_count = 229`, and `migration_ready = false`.
- The only reported migration blocker is `fallback-authored candidate paths: 8335`.
- The largest fallback residuals are concentrated in `kcs/sound/kc9998`, furniture families, BGM, map/gauge resources, `useitem/card*`, `titlecall_*`, and area/world-select style UI families.
- `main-decoder/out/resources/audio_resources.json` currently reports zero categorized BGM IDs and no explicit voice-file groups.
- `main-decoder/out/resources/ui_resources.json` currently reports zero map files, zero useitem IDs, zero area IDs, and zero world-select files.
- `main-decoder/out/resources/resource_id_sets.json` still leaves the sparse ship/slot subset families unresolved.

That means the next leverage point is to improve decoder extraction fidelity for the families that still dominate fallback, then tighten Rules-path fallback so newly covered members stop being primarily generated by Rust-owned lists.

## Goals / Non-Goals

**Goals:**

- Raise decoder bundle coverage for the highest-value residual families currently keeping `migration_ready` false.
- Make decoder-authored sound rules materially cover observed `kc9997`, `kc9998`, and `kc9999` bucket members instead of leaving those families mostly to fallback generation.
- Make Rules-path residual fallback operate at the uncovered-member level, not at the whole-family level, for the families this change improves.
- Keep progress measurable through `decoder_cachelist_compare` so fallback-count reduction is the primary success metric.

**Non-Goals:**

- Do not switch the default bootstrap CLI from `Default` or `Manifest` to `Rules`.
- Do not eliminate every remaining fallback path in one change.
- Do not remove `main-decoder/src/path-rules.ts` or attempt to complete the broader “decoder is the only semantic authority” cleanup in this round.
- Do not expand this change into gameplay traits, `_impl` gameplay functions, SeaORM entities, or KCSAPI route handlers.
- Do not chase broad `kcs2/img/*` versioned-art migration unless a specific decoder bundle gap in scope requires it.

## Decisions

### D1: Treat decoder bundle gaps as the primary migration blocker, not CLI or plumbing

This change will target the decoder output families that dominate current fallback counts rather than spending effort on more pipeline plumbing or strategy-default productization.

Priority families are:

- sound-rule families: `kc9997`, `kc9998`, `kc9999`, titlecall-related explicit audio, and categorized BGM
- UI/resource families: map, gauge, furniture, useitem card/underline, area, and world-select
- sparse ship/slot subsets still contributing to the remaining `candidate-only` precision tail

Rationale:

- The current comparison report already proves the decoder-first pipeline can reproduce the entire baseline set.
- The main missing property is ownership: too many paths still come from fallback-authored stages.
- Productizing `Rules` before reducing `8335` fallback-authored paths would harden a pipeline that is still semantically too dependent on Rust-owned lists.

Alternative considered: add a `--rules` CLI path or switch defaults first, then improve decoder coverage incrementally. Rejected because the migration report already says the candidate is not ready.

### D2: Improve decoder extraction against real decoded patterns, not only narrow regex fixtures

`main-decoder` extraction work will focus on the real decoded module graph patterns that currently leave `audio_resources.json`, `ui_resources.json`, and `resource_id_sets.json` mostly empty.

Concretely:

- `main-decoder/src/audio-resources.ts` will grow extraction for observed BGM/category tables, titlecall ranges, and explicit voice-file groups that appear in decoded modules but are not matched by the current narrow patterns.
- `main-decoder/src/ui-resources.ts` will grow extraction for explicit map file groups, useitem card/underline IDs, area IDs, world-select files, and richer furniture groups.
- `main-decoder/src/resource-id-sets.ts` and `main-decoder/src/cache-rules.ts` will be extended where sparse subset or sound bucket coverage is currently left unresolved despite decoder-observable evidence.

Rationale:

- The current extractor tests prove basic toy examples, but the live output still shows near-empty bundle assets for the families that matter.
- The next change should optimize for real pipeline output, not just unit-level pattern matching.

Alternative considered: patch Rust fallback lists first while leaving decoder outputs sparse. Rejected because it would reduce the visible blocker without improving actual decoder authority.

### D3: Tighten Rules-path fallback at the member level for newly covered families

As decoder outputs become stronger, `crates/emukc_bootstrap/src/make_list/source/kcs/mod.rs` and `crates/emukc_bootstrap/src/make_list/source/kcs2/resources/*.rs` will be updated so Rules-path fallback fills only the residual uncovered members of a family, not the whole family unconditionally.

This matters most for:

- `kcs/sound/*` bucket families
- furniture families that already have explicit decoder-observed subsets or paths
- map/gauge/useitem/area/world-select families where decoder bundle assets now enumerate concrete members

Rationale:

- Today Rules-path attribution remains high partly because whole-family fallback still runs after rule-authoritative generation.
- Once decoder bundle assets can enumerate concrete members, residual fallback should be explicit and narrow.

Alternative considered: leave Rules-path fallback behavior unchanged and rely only on compare reporting. Rejected because that would preserve false migration debt even after decoder bundle quality improves.

### D4: Use compare reports as the acceptance gate for this change

This change will treat `.data/decoder_rules_compare_current.json` as the baseline and require a regenerated report showing material reduction in fallback-authored paths for the targeted families.

Success indicators:

- `baseline_only_count` remains zero
- `fallback_authored_candidate_count` drops materially from `8335`
- `sound_fallback_authored_candidate_count` drops materially from its current level
- targeted top residual prefixes shrink or disappear
- the `candidate-only` ship/slot precision tail does not regress and ideally shrinks

Rationale:

- The project already has the right comparison loop.
- The decoder bundle gap work is valuable only if it measurably reclaims ownership from fallback.

Alternative considered: accept implementation completeness based on unit tests only. Rejected because bundle-quality work must be validated against the real candidate-versus-baseline report.

## Risks / Trade-offs

- [Risk] The decoded `main.js` may not expose enough stable evidence to fully close some targeted families. → Mitigation: preserve partial/unresolved states explicitly and scope fallback suppression only to families with newly proven coverage.
- [Risk] Better decoder extraction can increase `candidate_only_count` if family boundaries are over-expanded. → Mitigation: keep compare-based verification and treat the current `229` candidate-only tail as a precision guardrail.
- [Risk] Sound extraction work may conflate explicit audio coverage with algorithmic `kcs/sound/*` rule coverage. → Mitigation: keep `audio_resources.json` and sound rules distinct, and continue using the existing sound-domain breakdown in comparison reports.
- [Risk] Rules-path fallback suppression could hide real missing members if decoder bundle assets are incomplete. → Mitigation: apply suppression only where the decoder bundle now enumerates concrete members or covered subsets, and keep residual reporting explicit.
- [Risk] The old archived task lists for decoder coverage are stale and may no longer reflect current code reality. → Mitigation: drive this change from live emitted bundle contents and current compare residuals instead of archived checkbox state.

## Migration Plan

1. Strengthen decoder extraction for the highest-value audio/UI/sparse-subset families and regenerate `main-decoder/out/resources/*`.
2. Update Rust Rules-path generation to consume the improved bundle members as authoritative where concrete coverage now exists.
3. Regenerate comparison reports with `decoder_cachelist_compare` and inspect targeted residual families.
4. Keep CLI defaults unchanged; use the report delta to decide whether a later `Rules` productization/default-switch change is justified.

Rollback strategy:

- Revert decoder extractor changes and regenerated assets if the new extraction meaningfully increases over-generation.
- Revert any Rules-path fallback narrowing that incorrectly suppresses required baseline paths.
- Because this change does not switch user-facing defaults, rollback remains internal to decoder-first candidate generation.

## Open Questions

- For titlecall coverage, do we want the decoder bundle to preserve explicit reachable file counts or preserve category-plus-range semantics and let Rust expand them?
- For map and gauge families, should decoder output enumerate concrete file names directly, or should it preserve higher-level event/default structure plus a deterministic expansion rule?
- For `kc9998` and other sound buckets, is the missing coverage primarily an extractor gap, or does some of it depend on runtime-loaded data outside decoded `main.js`?
- How much of the remaining `candidate-only` ship/slot tail should be folded into this change versus held for a dedicated precision-cleanup follow-up?
